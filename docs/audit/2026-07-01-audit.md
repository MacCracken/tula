# tula — Security Audit (2026-07-01)

**Scope**: the tula codec at `[Unreleased]` (M0–M4 + this pass). Reviewed:
`src/format.cyr`, `src/sign.cyr`, `src/dtype.cyr`, `src/fileio.cyr`,
`src/error.cyr`. Method: re-derive each invariant from the code (not the
comments), then diff against the format contract. Six dimensions.

**Threat model** (see `SECURITY.md`): the **reader** is the trust boundary — it
loads *foreign* checkpoints and must never OOB-deref, overflow, or crash on a
lying/truncated/hostile file. The **writer** (builder/serialize) is the caller's
own trusted code. Authenticity is `sigil`'s (Ed25519); tula validates *structure*.

**Verdict**: **PASS.** No memory-safety or overflow defect found on the reader
path. One over-read sharp-edge on the *consumer* side (shape-vs-`data_len`) was
closed this pass with `tula_entry_payload_ok`. Fuzzed 2,003,000 iterations clean.

---

## 1. Memory safety

- **Reader accessors are gated by `tula_open` → `tula_validate`** ([format.cyr]).
  Validation proves, before any accessor runs: entries live in
  `[manifest_off, payload_off) ≤ len` (so each 144-byte entry read is in-bounds),
  and every tensor's `payload_off + data_off + data_len ≤ len`. Therefore
  `tula_entry`, `tula_entry_data`, `tula_entry_dim`, `tula_find`, `tula_is_signed`
  cannot read outside `[buf, buf+len)`. Confirmed by the fuzzer's `check_opened`
  invariant (every accepted buffer re-validates and every payload range is
  in-bounds) over 2M iterations — **0 breaches**.
- **`tula_verify`** reads `[0, sig_off)` and the 64-byte sig at `sig_off`; the
  validator pins `sig_len == 64` and `sig_off + 64 ≤ len`, so the fixed 64-byte
  read is always in-bounds (gap found + closed in M4 — see §4).
- **dtype pack/unpack** (`dtype.cyr`) are pure caller-driven transforms; they
  read/write exactly what the caller's `n`/buffer sizes say. On the *reader* side
  a lying shape could over-drive `n` — see §3 (Finding F-1, fixed).
- **Writer path** (`tula_builder_add` / `tula_serialize`) trusts the caller
  (its own code); `memcpy` copies caller-declared `data_len` from caller buffers.
  Not an untrusted boundary.

## 2. Integer overflow

- **Validator arithmetic is overflow-safe by construction**, bounded by the
  resource caps (§5): `count ≤ 2²⁰` ⇒ `count*ENTRY_SIZE(144) < 2²⁸`;
  `payload_len ≤ 2⁴⁰`; all bounds checks use the *subtraction* form
  (`data_len > payload_len - data_off` with `data_off ≤ payload_len` first), so
  no sum can wrap. `content_len = payload_off + payload_len ≤ len ≤ 2⁴⁰+…` stays
  far below 2⁶³.
- **`tula_hdr_total`** (pre-alloc, untrusted header) caps each of
  `payload_off/len`, `sig_len` at `2⁴⁰` *before* summing, and returns -1 on any
  negative — the sum can never overflow, and the readers reject `total < 64` or
  `> cap`.
- **`tula_entry_payload_ok`** guards its dim product with
  `n > TULA_READ_MAX_BYTES / d` before multiplying — an overflowing shape returns
  0, no wrap (tested).
- **Writer**: `tula_serialize` computes `total` from caller-accumulated sizes with
  no overflow guard — a caller adding tensors whose aligned sizes sum past 2⁶³
  would get a wrapped `alloc`. Documented as a **caller responsibility** (trusted
  writer); not reachable from untrusted input.

## 3. Untrusted-input bounds

- Every structural field is validated against `len`: magic, version,
  `manifest_off == 64`, `payload_off == 64 + count*ENTRY_SIZE ≤ len`,
  `payload_off + payload_len ≤ len`, per-entry `ndim ∈ [0,8]`, NUL-terminated
  name within 48, `data_off + data_len ≤ payload_len`, sig-field consistency.
  Rejections are exhaustively covered by `tests/tcyr/hardening.tcyr` (30) and the
  fuzzer.
- **Finding F-1 (fixed this pass, severity: low/consumer-side).** The dtype
  *unpack* helpers read a byte count derived from the element count `n` the
  **caller** passes. A foreign-checkpoint consumer that computes `n` from an
  untrusted `shape` (product of dims) and then unpacks could over-read, because
  `tula_validate` bounds `ndim` but **not dim values** — a file can declare
  `shape=[10⁹]` while storing only 2 payload bytes. The read would run past the
  tensor's `data_len` (and, for the last tensor, potentially past `len`).
  - *Resolution*: added `tula_packed_len(dtype, nelems)` and
    `tula_entry_payload_ok(e)` (`dtype.cyr`). A consumer runs
    `tula_entry_payload_ok(e)` (overflow-safe) before any unpack; it returns 0
    when `data_len` is smaller than the dtype+shape imply. Documented as required
    consumer discipline in `SECURITY.md` and `docs/api.md`. tula stays a codec —
    it *provides* the check; the consumer applies it, since only the consumer
    knows whether it is decoding shape-first or `data_len`-first. Tested in
    `dtype.tcyr` (consistent → ok; oversized shape → not ok; overflowing dim →
    not ok).
- **Non-finding**: unvalidated dim values do **not** affect tula's own accessors
  (they bound on `data_off`/`data_len`, never `shape`), so a huge shape cannot
  break the reader itself.

## 4. Signature-verify correctness

- **Range**: `tula_verify` calls `ed25519_verify(pk, r, sig_off, r + sig_off)` —
  the signed message is exactly `[0, sig_off)` (header incl. the tamper-evident
  `sig_off`/`sig_len` fields ‖ manifest ‖ payload), matching the sign path
  `ed25519_sign(sk, buf, sig_off, buf + sig_off)`. The signature is **not** in the
  signed range (it sits at `sig_off`). Correct.
- **sigil primitive** (`sigil/src/ed25519.cyr`): verification rejects
  non-canonical `S ≥ L` (RFC 8032 §8.4 — anti-malleability) and finishes with
  `ct_eq_bytes_lens(R_bytes, 32, sig, 32)` — a constant-time compare. No
  data-dependent early-out on the comparison.
- **Timing**: signature *verification* processes only public data (pk, message,
  signature) — there is no secret, so verify timing is not a side-channel of
  concern. Secret-key timing lives entirely in `ed25519_sign` (sigil's
  constant-time scalar-mult), which tula calls but does not implement.
- **Malleability at the tula layer**: `sig_len` is pinned to `{0, 64}` and
  `sig_off == content_len`, so there is exactly one valid signature placement/size
  for a given content — no room to smuggle extra bytes into a "valid" signed file.

## 5. Resource caps

- `TULA_READ_MAX_TENSORS = 2²⁰` (rejects absurd manifest counts; also the overflow
  bound for `count*ENTRY_SIZE`) and `TULA_READ_MAX_BYTES = 2⁴⁰` (1 TiB total-size
  ceiling). `tula_read_file` / `tula_open_mmap` apply the ceiling via
  `tula_hdr_total` **before** allocating or mmap'ing, so a tiny file that lies
  "`payload_len = 10¹⁸`" is rejected up front — no bomb alloc. `alloc == 0` is
  checked in `tula_read_file`.
- These caps are generous for real checkpoints yet bound all validator arithmetic
  well inside i64. A future consumer needing >1 TiB files raises the cap
  deliberately (a format-level decision), not by accident.

## 6. Fail-loud on precondition breach

- `tula_open` returns `0` on *any* validation breach — never a partially-valid
  handle. The granular reason is available via `tula_validate` (a specific
  `TULA_ERR_*` with a name in `tula_err_name`). `tula_read_file` /
  `tula_open_mmap` return `0` on short read, bad magic, oversized total, failed
  alloc/mmap. No silent truncation or best-effort parse anywhere on the reader
  path.
- `tula_verify` returns a three-state result (`UNSIGNED` / `OK` / `BAD`) — an
  unsigned file is never silently treated as verified; a bad signature is never
  silently accepted.

---

## Actions taken this pass

- Closed **F-1**: `tula_packed_len` + `tula_entry_payload_ok` (`dtype.cyr`), tests
  in `dtype.tcyr`.
- (M4, prerequisite) pinned `sig_len ∈ {0, 64}` in the validator so `tula_verify`'s
  fixed read is provably in-bounds; test in `hardening.tcyr`.

## Residual / documented-as-caller-owned

- Writer-side overflow on absurd accumulated sizes (`tula_serialize`) — caller
  (trusted-writer) responsibility.
- Consumers **must** call `tula_entry_payload_ok(e)` before unpacking foreign
  tensors, and supply/validate their own keys — tula owns no key management.
